Создаём основу игрового сервера
В этом уроке вы начнёте работу над многопользовательской игрой “Dog Story”. 
Механика игры
Игрок управляет псом, задача которого — находить потерянные вещи и относить их в ближайшее бюро находок. На карте одновременно присутствуют несколько псов, которыми управляют другие пользователи, подключившиеся к серверу. Цель игры — набрать как можно больше игровых очков, которые начисляются за доставленные в бюро находок вещи.
На стартовом экране отображается список карт. После выбора карты открывается экран игры.
На этом экране отображается фрагмент карты города. Карта состоит из дорог, по которым перемещаются псы игроков. На дорогах время от времени появляются потерянные вещи, которые нужно подобрать раньше противников и отнести в ближайшее бюро находок. Псы, управляемые игроками, могут нести не более трёх предметов за раз, поэтому регулярно нужно наведываться в офис. Кроме дорог и офисов, на карте располагаются здания, которые несут исключительно декоративную функцию и не влияют на ход игры.
В информационной области игрового экрана отображаются предметы, которые несёт пёс игрока, а также набранные очки.
Архитектура игры
Игра будет выполнена в виде клиент-серверного приложения:

    Сервер отвечает за хранение и обновление игрового состояния, отправку состояния клиентам, обработку команд управления собаками. Размещение игрового кода на стороне сервера поставит игроков в одинаковые условия и уменьшит возможность жульничества путём модификации кода, выполняемого на стороне клиента.
    Клиент принимает ввод пользователя и отправляет его на сервер. Также клиент получает актуальное игровое состояние с сервера и отображает его пользователю. Клиентская часть игры (фронтенд) будет выполняться в веб-браузере и использовать технологии HTML/CSS/JavaScript.

Код клиентской части игры у вас будет, а серверную часть вы разработаете самостоятельно.
Игровой сервер будет предоставлять клиентам REST API. Он поможет получить доступ к текущему состоянию игры и управлять движением своего курьера.
REST API — это архитектурный стиль взаимодействия компонентов распределённого приложения в сети. В основе REST API (его ещё называют RESTful API) лежат 6 принципов написания распределённых приложений:

    Отделение клиента от сервера. Это позволяет разделить зоны ответственности клиента и сервера. Например, хранение данных входит в зону ответственности сервера, что позволяет полностью изменить способ хранения и обработки данных на сервере, не затрагивая код клиентов. Формирование интерфейса пользователя, напротив, является зоной ответственности клиента
    Отсутствие состояния. В период между запросами никакая информация о состоянии клиента на сервере не хранится. Каждый запрос от клиента должен быть составлен так, чтобы сервер получил всю необходимую информацию для выполнения запроса. Это облегчает задачу распределения нагрузки между несколькими серверами.
    Кэширование. Ответы сервера должны иметь явное или неявное обозначение как кэшируемые или некэшируемые. Так клиенты не будут получать устаревшие или неверные данные в ответ на последующие запросы.
    Единый интерфейс. Ресурсы сервера используют URI в роли идентификаторов ресурсов. Формат хранения ресурсов на сервере отделён от представления, в котором ресурсы отдаются клиенту. Это означает, что сервер отдаёт ресурсы клиенту в некотором стандартном формате вроде JSON или XML, а хранит их в удобном для обработки виде, например, в базе данных.
    Слои. Обычно клиент не может определить, взаимодействует он с конечным сервером или с промежуточным. За счёт этого промежуточные серверы могут улучшить масштабируемость системы за счёт балансировки нагрузки или кэширования.
    Отправка кода по требованию. Этот принцип позволяет настраивать или расширять функциональность клиентов, отправляя им код, который предназначен для выполнения на стороне клиента, например, в виде скриптов. Это опциональная возможность, для вашего сервера она не понадобится.

Следование этим принципам помогает строить масштабируемые веб-приложения.
Обзор серверной части игры
Задачи сервера:

    Хранение и управление состоянием игры. Состояние игры включает в себя данные о картах и игроках, подключённых к игре. На сервере будет реализована игровая логика: правила перемещения игроков и их взаимодействия с объектами на карте, начисление баллов.
    Отдача статических файлов необходимых для работы игры в браузере: изображения, код на HTML/CSS/JavaScript.
    Обработка клиентских запросов: получение информации об игровом состоянии, получение команд управления псами.

Вы начнёте с сервера с простым функционалом и будете постепенно добавлять новые возможности.

    При старте ваш сервер должен загрузить конфигурационный файл с информацией об имеющихся картах. То, что описание игры хранится отдельно от кода игры, сделает игровой сервер более гибким: для настройки игры не понадобится вносить изменения в код на C++ и перекомпилировать программу.
    Загрузив конфигурационный файл, сервер должен предоставить REST API, чтобы клиенты могли получить список карт и подробную информацию о каждой игровой карте.

Работа с JSON
Как формат конфигурационных файлов сервер будет использовать JSON (JavaScript Object Notation). Это распространённый текстовый формат обмена данными, основанный на JavaScript. Чтобы обмениваться данными с клиентами, также будет использоваться JSON. Это облегчит написание клиентского приложения на JavaScript, в котором есть встроенная поддержка такого формата.
Для работы с JSON на C++ существует множество библиотек. Не стала исключением и библиотека Boost — в её состав входит библиотека Boost.Json. Чтобы подключить эту библиотеку, подключите файл <boost/json.hpp>. Также понадобится подключить файл <boost/json/src.hpp> в одном и только одном из .cpp-файлов вашего проекта. Такой способ подключения самый простой и не требует отдельной сборки библиотеки (хотя и это легко сделать, используя conan).

#include <boost/json.hpp>

// Этот заголовочный файл надо подключить в одном и только одном .cpp-файле программы
#include <boost/json/src.hpp>
#include <iostream>
#include <string>

namespace json = boost::json;
using namespace std::literals;

int main() {
    auto json_str = R"({"name": "Harry Potter"})";
    auto value = json::parse(json_str);
    std::cout << value.as_object().at("name") << std::endl; // Harry Potter
    std::cout << json::serialize(value) << std::endl; // {"name": "Harry Potter"}
} 

Распределение кода сервера по модулям
В коде сервера можно выделить несколько самостоятельных модулей:

    Модель игры (model). Содержит классы и функции, отвечающие за хранение информации об игровых объектах, а также игровую логику.
    Каркас HTTP-сервера (http_server). Отвечает за асинхронный обмен данными с клиентами. Делегирует обработку запросов внешнему коду. Код этого модуля вы собрали в прошлом уроке.
    Обработчик запросов (request_handler) — содержит код, отвечающий за обработку HTTP-запросов клиентов.
    Загрузчик модели игры из JSON при старте сервера (json_handler).
    Точка входа в приложение и сборка компонентов серверного приложения воедино (main).

На рисунке приведена примерная схема модулей приложения и зависимостей между ними. Сами модули условно разделены на несколько слоёв. Такое разделение на слои помогает разграничить зоны ответственности модулей в приложении, облегчить разработку и тестирование.

*Архитектурная схема приложения: зависимости между модулями и распределение модулей по слоям *

    Модель не имеет зависимостей от нижележащих модулей. Это значит, что при изменении нижележащих модулей код модели не изменится. Например, если вы решите заменить Boost.Json на другую библиотеку, изменять код модели будет не надо.
    Слой инфраструктуры содержит код, соединяющий приложение с сетевыми сервисами, слоем хранения данных. Код на этом слое зависит от внешних библиотек, таких как Boost.Json, Boost.Asio и Boost.Beast.
    Модуль main содержит точку входа в приложение (функцию main). Его задача — создать основные компоненты системы, соединить их между собой.

При разработке приложения вы можете разделить код приложения по модулям и задать зависимости между ними иным способом. Но лучше избегать зависимостей модели от инфраструктурного кода.
Задание
Разработайте веб-сервер, предоставляющий клиентам REST API со списком карт. Вы найдёте заготовку кода игрового сервера в каталоге sprint1/problems/map_json/precode.
Формат входных данных
Программе через единственный параметр командной строки подаётся путь к конфигурационному файлу в формате JSON, содержащему информацию об игровом поле.
Следующая команда запускает сервер и передаёт ему файл config.json:

game_server config.json 

Файл содержит JSON-объект с единственным свойством maps. Это свойство описывает массив игровых карт или уровней.

{
    "maps": []
} 

Каждый элемент этого массива — объект, описывающий дороги, здания и офисы на игровой карте. Свойства этого объекта:

    id — уникальный идентификатор карты. Тип: строка.
    name — название карты, которое выводится пользователю. Тип: строка.
    roads — дороги игровой карты. Тип: массив объектов. Массив должен содержать хотя бы один элемент.
    buildings — здания. Тип: массив объектов. Массив может быть пустым.
    offices — офисы бюро находок. Тип: массив объектов. Массив может быть пустым.

В массиве roads содержатся элементы двух типов, каждый из которых описывает горизонтальный либо вертикальный отрезок дороги. Порядок следования элементов в этом массиве может быть произвольным.
Горизонтальный отрезок дороги — объект со свойствами x0, y0, x1, задающих целочисленные значения координат начала и конца отрезка. Пример:

{ "x0": 0, "y0": 0, "x1": 40 } 

Вертикальный описывается объектом со свойствами x0, y0, y1, которые задают целочисленные значения координат начала и конца отрезка дороги. Пример:

{ "x0": 0, "y0": 0, "y1": 20 } 

Здание на карте представлено в виде объекта со свойствами x, y, w, h, которые задают целочисленные координаты верхнего левого угла дома, его ширину и высоту.
Пример:

{ "x": 5, "y": 5, "w": 30, "h": 20 } 

Офис представлен объектом со свойствами:

    id — уникальный идентификатор офиса. Строка.
    x, y — координаты офиса. Целые числа. Офисы располагаются на дороге, так как собаки могут перемещаться только по дорогам.
    offsetX, offsetY — смещение изображения офиса относительно координат x и y. Эти параметры используются только на стороне клиента для визуализации офиса и на игровую логику не влияют.

Пример описания офиса:

{
  "id": "o0",
   "x": 40,
   "y": 30,
   "offsetX": 5,
   "offsetY": 0
} 

Пример конфигурационного файла:

{
  "maps": [
    {
      "id": "map1",
      "name": "Map 1",
      "roads": [
        { "x0": 0, "y0": 0, "x1": 40 },
        { "x0": 40, "y0": 0, "y1": 30 },
        { "x0": 40, "y0": 30, "x1": 0 },
        { "x0": 0, "y0": 0, "y1": 30 }
      ],
      "buildings": [
        { "x": 5, "y": 5, "w": 30, "h": 20 }
      ],
      "offices": [
        { "id": "o0", "x": 40, "y": 30, "offsetX": 5, "offsetY": 0 }
      ]
    }
  ]
} 

В конфигурационном файле содержится одна карта с названием “Map 1” и id, равным “map1”. Карта состоит из 4 отрезков дороги, 1 здания и 1 бюро находок. Изображение бюро смещено на 5 единиц вправо относительно координат офиса (40, 30).

Схематичное представление карты
Описание поведения веб-сервера
Сервер должен принимать HTTP-запросы на порт 8080. В этом задании вам нужно реализовать поддержку запросов двух типов:

    получение списка карт,
    получение описания карты с заданным id.

/api/v1/maps — GET-запрос возвращает в теле ответа краткую информацию обо всех картах в виде JSON-массива объектов с полями id и name. Ответ должен иметь обязательный заголовок Content-Type со значением application/json. Карты в ответе должны быть перечислены в том же порядке, что и в конфигурационном файле.
Пример тела ответа:

[{"id": "map1", "name": "Map 1"}] 

/api/v1/maps/{id-карты} — GET-запрос возвращает в теле ответа JSON-описание карты с указанным id, семантически эквивалентное представлению карты из конфигурационного файла. Ответ сервера должен иметь заголовок Content-Type со значением application/json. 
Пример тела ответа на запрос GET /api/v1/maps/map1 HTTP/1.1:

{
  "id": "map1",
  "name": "Map 1",
  "roads": [
    { "x0": 0, "y0": 0, "x1": 40 },
    { "x0": 40, "y0": 0, "y1": 30 },
    { "x0": 40, "y0": 30, "x1": 0 },
    { "x0": 0, "y0": 0, "y1": 30 }
  ],
  "buildings": [
    { "x": 5, "y": 5, "w": 30, "h": 20 }
  ],
  "offices": [
    { "id": "o0", "x": 40, "y": 30, "offsetX": 5, "offsetY": 0 }
  ]
} 

Обработка ошибок
В случае ошибки сервер должен возвращать ответ с соответствующим HTTP-кодом ошибки. Заголовок Content-Type ответа сервера должен иметь значение application/json. Тело ответа — JSON-объект с полями code и message, содержащими код ошибки и её понятное описание.
Если в запросе на получение описания карты указан id несуществующей карты, сервер должен вернуть ответ со статус-кодом 404 и следующим телом ответа:

{
  "code": "mapNotFound",
  "message": "Map not found"
} 

Поле message может содержать и другой понятный человеку текст, однако поле code должно иметь значение “mapNotFound”.
Если URI-строка запроса начинается с /api/, но не подпадает ни под один из текущих форматов, сервер должен вернуть ответ с 400 статус-кодом. Тело ответа должно содержать такой JSON-объект:

{
  "code": "badRequest",
  "message": "Bad request"
} 

Здесь поле message также может иметь другой понятный текст, однако поле code должно иметь значение “badRequest”.
Обработка сигналов
После своего старта сервер должен реагировать на сигналы SIGINT и SIGTERM и корректно завершать свою работу при получении этих сигналов.
Как будет тестироваться ваш код
В репозитории будет набор тестов, которые запустят ваш сервер, передав ему путь к конфигурационному файлу с описанием нескольких карт. Затем тест отправит несколько запросов к серверу и сравнит ответ сервера с ожидаемым.
Как только сервер будет готов обрабатывать запросы, он должен вывести в stdout строку Server has started..., чтобы тесты не начали проверять его работу раньше времени.
При тестировании проект будет собран утилитой CMake и запущен. Не меняйте название главной цели — game_server.

Подсказки

    Используйте фрагменты кода асинхронного сервера, в частности модуль http_server.
    В заготовке кода есть классы и структуры, описывающие основные сущности модели игры. Вы можете использовать их либо реализовать свою модель.
    Перед вызовом функции RunWorkers выведите надпись Server has started.... Так тесты поймут, что сервер проинициализирован и готов обрабатывать запросы.

int main() {
    ...
    http_server::ServeHttp(ioc, {address, port}, [](auto&& req, auto&& sender) {
        ...
    });

    // Эта надпись сообщает тестам, что сервер запущен и готов обрабатывать запросы
    std::cout << "Server has started..."sv << std::endl;

    RunWorkers(num_threads, [&ioc] {
        ioc.run();
    });
}
