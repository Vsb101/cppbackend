Отдача статических файлов
Ваш игровой сервер предоставляет REST API для доступа к игровым картам. Однако поиграть пока не выйдет, так как нет игрового клиента, который запускается прямо в браузере. В этом уроке вы научите сервер отдавать статические веб-страницы в формате HTML, сценарии на JavaScript, каскадные таблицы стилей CSS, а также файлы изображений.
Когда вы дважды щелкаете мышью по иконке файла на компьютере, операционная система запускает подходящую для этого файла программу: видеоплеер, текстовый редактор, редактор электронных таблиц или программу для просмотра изображений.
Как выбирается подходящая программа, зависит от вашей ОС: в Windows используется расширение файла, а в Linux дополнительно может анализироваться его содержимое. При установке программа может зарегистрировать себя в системе как способную открывать файлы определённых типов.
Современный веб-браузер ведёт себя как операционная система в миниатюре — в нём можно смотреть изображения и видео, редактировать документы. Когда вы вводите URL в адресной строке, браузер делает GET-запрос к серверу по указанному адресу. После получения ответа обрабатывает его. Так же, как операционная система выбирает, какой программой открыть файл, браузер должен определить, как обработать ответ сервера: 

    отобразить в виде простого текста,
    воспроизвести как медиа-файл,
    показать как изображения.

Медиа-типы
Первое, чем руководствуется браузер — смотрит на содержимое заголовка Content-Type в ответе сервера. Этот заголовок сообщает браузеру MIME-тип (англ. MIME-type) контента. MIME — стандарт, который позволяет описать передачу различных типов данных по электронной почте и передачу сообщений через интернет. MIME-типы также называют медиа-типами (англ. media type).
Медиа-тип — это строка, состоящая из двух частей: типа и подтипа, разделённых косой чертой “/” без пробела. Например: text/plain.
Тип определяет категорию, к которой относятся данные: аудио, видео, текст, изображения. Подтип конкретизирует формат данных. Например, и text/html, и text/plain служат для представления текстовой информации, однако первый из них поддерживает HTML-разметку, в которой символы <, > и & имеют особое значение, а второй разметку не поддерживает.
К MIME-типу через точку с запятой может быть добавлена вспомогательная информация в формате параметр=значение.  Например, для текста может быть указана кодировка символов:
text/plain; charset=UTF-8
Используя информацию о медиа-типе, браузер может корректно определить тип полученного содержимого и правильно отобразить его пользователю.
Выделяются следующие базовые типы данных:

    application — любой тип данных, который не попадает в остальные категории. Как правило, для обработки этих данных требуется поддержка со стороны приложения. Например: application/json, application/pdf.
    audio — аудиоданные. Например: для mp3-файлов используется audio/mpeg.
    example. Зарезервирован для использования в документации. Пример: audio/example.
    font. Шрифты, например font/woff, font/ttf, font/otf.
    image — растровые или векторные изображения. Могут быть как статическими, так и анимированными. Например: image/png, image/jpeg, image/gif.
    model — описания трехмерных сцен и моделей. Например: model/vrml.
    text — текстовые данные в понятном человеку формате. Примеры: text/plain, text/html, text/csv.
    video — видеоданные. Например: video/mp4, video/webm.

Для двоичных данных неизвестного типа используется медиа-тип application/octet-stream. Получив содержимое в этом формате, браузеры не пытаются его интерпретировать, а предлагают пользователю сохранить файл на диск. Это происходит, например, когда вы нажимаете кнопку «скачать» в веб-интерфейсе электронной почты.
Для текстовых данных неизвестного формата используется по умолчанию тип text/plain. Встретив такое содержимое, браузер попытается отобразить его в виде текста.
При отсутствии заголовка Content-Type браузер может попытаться определить MIME-тип по косвенным признакам, например по расширению «файла» в URL-адресе или по его содержимому. Браузеры так делают, чтобы корректно отобразить пользователю веб-страницу, даже когда веб-сервер некорректно сконфигурирован. Чего не сделаешь ради удобства пользователя. Эта технология называется MIME sniffing (буквально: разнюхивание MIME-типа) и реализована в разных браузерах по-разному.
Размещение ресурсов на сервере
В протоколе HTTP клиент не имеет доступа к файловой системе сервера. Вместо этого клиент оперирует понятием «ресурс». Путь к ресурсу задаётся в URI-строке запроса. Например, так выглядит запрос ресурса с идентификатором /index.html: 

GET /index.html HTTP/1.1 

Как именно будет сформировано содержимое ресурса с идентификатором /index.html, решает сервер. Для клиента нет никакой возможности узнать, прочитал сервер данные с диска, считал из базы данных, запросил их у другого сервера или сгенерировал на лету.
Ваш сервер игры “Dog Story” предоставляет REST API, поддерживающее два типа запросов:

    GET /api/v1/maps для получения списка карт в формате JSON,
    GET /api/v1/maps/{map_id} для получения карты с идентификатором {map_id}.

Данные, возвращаемые в ответ на эти запросы, ваш сервер генерирует на лету. Для этого он сериализует модель игрового мира, загруженную при старте сервера, в JSON-представление и возвращает результат в виде текста с Content-Type: application/json.
Чтобы сервер мог отдавать клиентам файлы, надо научить его по URI-строке определять путь к файлу на диске и возвращать содержимое этого файла.
Какой из перечисленных ниже способов преобразовать URI-строку в путь к файлу на диске следует выбрать для вашего сервера:
Использовать URI-строку в качестве абсолютного пути к файлу.
Использовать URI-строку в качестве пути относительно домашней директории текущего пользователя.
Правильный ответ
Использовать URI-строку в качестве пути внутри директории, играющей роль корневого каталога. Итоговый путь ни в коем случае не должен выходить за пределы этой директории.
Так пользователи из интернета будут иметь доступ на чтение лишь к небольшому подмножеству файлов вашего сервера, например, к файлам HTML/CSS/JavaScript и изображениям.
Как вам задача?
Обычно для размещения статических файлов на сервере создаётся специальный каталог, содержимое которого сервер отдаёт своим клиентам. Путь к этому каталогу задаёт корневую директорию сервера. Все пути в этом каталоге задаются относительно этого каталога.
Здесь каталог wwwroot — корневой каталог, в котором размещаются файлы, которые сервер отдаёт клиентам. Внутри веб-страницы index.html можно ссылаться на другие файлы как по относительным, так и по абсолютным путям. Первый символ абсолютного пути — косая черта /.
Так как сервер предоставляет ещё REST API, при обработке запроса он должен определить, куда ссылается URI-строка запроса: на путь к файлу или точку входа в API. Сервер может использовать такую логику: если URI-строка запроса начинается с /api/, значит следует обработать запрос к API, в противном случае вернуть содержимое файла внутри корневого каталога (в приведённом выше примере — внутри каталога /home/game_server/wwwroot).

Как вам задача?
Метод request::target задаёт или возвращает string_view, ссылающийся на URI-строку запроса. Эта строка может содержать URL-кодированные (англ. URL-encoded) символы.
URL-encoding или percent-encoding — это способ кодирования данных, чтобы их можно было размещать внутри URI-строки с использованием ограниченного набора символов кодировки ASCII. В этой кодировке любой байт может быть представлен в виде трёх символов: служебного символа “процент” %, за которым следуют две шестнадцатеричные цифры, задающие значение байта.
Например, английская буква “G” имеет код символа 71 (0x47 в шестнадцатиричной системе счисления) и представляется в кодировке URL-encoding как %47. В URI некоторые символы — служебные, поэтому когда они используются в качестве данных, они всегда должны быть закодированы. Вот эти символы:

!    #    $    &    '    (    )    *    +    ,    /    :    ;    =    ?    @    [    ] 

В target-строке запроса должен быть закодирован символ «пробел», так как в первой строке запроса он отделяет target-строку от метода и версии протокола. Пробел может быть закодирован двумя способами: %20, а также символом +. Прочие символы могут быть представлены как в кодированном, так и обычном виде. Например, строки “HELLO%20WORLD”, “HELLO+WORLD” и “H%45LLO%20WORLD” — это одна и та же строка “HELLO WORLD”. В третьем случае символ E с кодом 69 (или 0x45 в шестнадцатеричной системе счисления) закодирован последовательностью %45.

Доступ к файлам и вопросы безопасности
В стандартной библиотеке для работы файловой системой служит библиотека <filesystem>. Класс std::filesystem::path предоставляет операции для манипуляции путями файлов. Например:

#include <filesystem>
#include <iostream>

using namespace std::literals;

int main() {
    namespace fs = std::filesystem;
    fs::path base_path{"/this/is/some/path/"s};
    fs::path rel_path{"../../another/path"s};
    fs::path abs_path = fs::weakly_canonical(base_path / rel_path);
    std::cout << abs_path << std::endl; // Выведет "/this/is/another/path"
} 

Используя классы и функции библиотеки std::filesystem, вы можете преобразовать URI-строку запроса в путь к файлу внутри корневого каталога веб-сервера. URI-строку нужно предварительно раскодировать, преобразовав последовательность символов %xx в соответствующий код символа. 
Когда конструируете путь к файлу, следите, чтобы результирующий путь находился внутри корневого каталога со статическими файлами. Иначе сервер будет в опасности: злоумышленник может заключить в URI-строку компоненты для перехода в родительский каталог (../) и получит содержимое файла с конфиденциальной информацией за пределами корневого каталога.

Проверить, что строка, возвращённая методом request::target, не содержит последовательность символов ... С её помощью вычислить абсолютный путь к файлу.

Сначала URI-декодировать строку, возвращённую методом request::target, и проверить, что она не содержит последовательность символов ../. С её помощью вычислить абсолютный путь к файлу.
Правильный ответ
Декодировать URI-строку, возвращённую методом request::target, с её помощью вычислить абсолютный путь к файлу и убедиться, что он находится внутри корневого каталога веб-сервера.
Как вам задача?
Отбраковав URI-строки, в которых содержится последовательность символов .., вы отбросите файлы с двумя подряд идущими точками в имени, например, file..name.txt. Недостаточно отбраковать и URI-строки с последовательностью ../. Строка, которую возвращает метод request::target, содержит символы в кодировке URL-encoding. В ней символы . и / могут быть закодированы в виде последовательностей %2E и %2F соответственно, поэтому поиск последовательности ../ не даст результатов.
Если сперва декодировать target-строку, а потом проверить в ней наличие ../, то невалидными будут считаться пути вроде /images/../index.html. В этом ложноположительном срабатывании большой проблемы нет, так как обычно браузеры такие запросы не делают. Гораздо серьёзнее то, что уязвимость сохранится при работе сервера в ОС Windows. В её файловой системе символом-разделителем каталогов кроме /  может быть ещё и символ \. Если злоумышленник передаст target-строку /%2E%2E%5C/secret.txt, то после декодирования получится путь <root>/..\secret.txt к файлу за пределами корневого каталога.
Простая защита — после URI-декодирования проверить , что абсолютный путь к файлу остаётся в пределах корневого каталога. Примените функцию std::filesystem::weakly_canonical, чтобы привести пути к каноничному виду (каноничный путь не содержит компонентов . и ..), и проверьте, что один путь — префикс другого:

#include <filesystem>
#include <cassert>

using namespace std::literals;
namespace fs = std::filesystem;

// Возвращает true, если каталог p содержится внутри base_path.
bool IsSubPath(fs::path path, fs::path base) {
    // Приводим оба пути к каноничному виду (без . и ..)
    path = fs::weakly_canonical(path);
    base = fs::weakly_canonical(base);

    // Проверяем, что все компоненты base содержатся внутри path
    for (auto b = base.begin(), p = path.begin(); b != base.end(); ++b, ++p) {
        if (p == path.end() || *p != *b) {
            return false;
        }
    }
    return true;
}

int main() {
    assert(IsSubPath("/root/path/sub-folder/file"s, "/root/path"s));
    assert(IsSubPath("/root/path/sub-folder/../file"s, "/root/path"s));
    assert(!IsSubPath("/root/path/../file"s, "/root/path"s));
} 

Поддержка файлов в Boost.Beast
Для поддержки передачи файлов в Boost.Beast служит класс boost::beast::http::file_body. Следующая программа создаёт HTTP-ответ с заголовком Content-Type: text/plain и телом с содержимым файла, который передаётся программе через первый параметр командной строки:

#include <boost/beast.hpp>
#include <iostream>

namespace beast = boost::beast;
namespace http = beast::http;
namespace sys = boost::system;

int main(int argc, char* argv[]) {
    using namespace http;
    using namespace std::literals;

    if (argc != 2) {
        std::cout << "Usage: "sv << argv[0] << " <file name>"sv << std::endl;
        return EXIT_FAILURE;
    }

    response<file_body> res;
    res.version(11);  // HTTP/1.1
    res.result(status::ok);
    res.insert(field::content_type, "text/plain"sv);

    file_body::value_type file;

    if (sys::error_code ec; file.open(argv[1], beast::file_mode::read, ec), ec) {
        std::cout << "Failed to open file "sv << argv[1] << std::endl;
        return EXIT_FAILURE;
    }

    res.body() = std::move(file);
    // Метод prepare_payload заполняет заголовки Content-Length и Transfer-Encoding
    // в зависимости от свойств тела сообщения
    res.prepare_payload();
} 

Метод message::prepare_payload заполняет заголовки Content-Length и Transfer-Encoding в зависимости от свойств тела сообщения. В этой программе для тела используется тип boost::beast::http::file_body, поэтому заголовок Content-Length будет равен размеру файла. Заголовок Transfer-Encoding будет пустым, так как тело сообщения передаётся в исходном виде и не использует никаких преобразований, таких как компрессия.
Библиотеки Boost.Asio и Boost.Beast поддерживают возможность расширения. Например, созданный запрос при помощи функции boost::beast::http::write можно записать не только в сокет, но и использовать любой объект, который содержит два метода write_some: первый возвращает ошибку через код возврата, а второй выбрасывает исключение. Проиллюстрируем эту возможность и создадим класс SyncWriteOStreamAdapter, адаптирующий std::ostream под нужды библиотеки Boost.Asio. Это позволит выводить объекты библиотеки Boost.Beast в stdout, что может быть полезно для отладки:

namespace net = boost::asio;

// Для синхронного вывода в Boost.Asio могут использовать любой тип, удовлетворяющий требованиям,
// описанным в документе:
// https://www.boost.org/doc/libs/1_78_0/doc/html/boost_asio/reference/SyncWriteStream.html
class SyncWriteOStreamAdapter {
public:
    explicit SyncWriteOStreamAdapter(std::ostream& os)
        : os_{os} {
    }

    template <typename ConstBufferSequence>
    size_t write_some(const ConstBufferSequence& cbs, sys::error_code& ec) {
        const size_t total_size = net::buffer_size(cbs);
        if (total_size == 0) {
            ec = {};
            return 0;
        }
        size_t bytes_written = 0;
        for (const auto& cb : cbs) {
            const size_t size = cb.size();
            const char* const data = reinterpret_cast<const char*>(cb.data());
            if (size > 0) {
                if (!os_.write(reinterpret_cast<const char*>(data), size)) {
                    ec = make_error_code(boost::system::errc::io_error);
                    return bytes_written;
                }
                bytes_written += size;
            }
        }
        ec = {};
        return bytes_written;
    }

    template <typename ConstBufferSequence>
    size_t write_some(const ConstBufferSequence& cbs) {
        sys::error_code ec;
        const size_t bytes_written = write_some(cbs, ec);
        if (ec) {
            throw std::runtime_error("Failed to write");
        }
        return bytes_written;
    }

private:
    std::ostream& os_;
};

    http::write(adapter, res);
} 

Запустив программу и передав ей в командной строке путь к файлу, содержащему строку Hello world, увидим:

HTTP/1.1 200 OK
Content-Type: text/plain
Content-Length: 11

Hello world 

Теперь всё готово к тому, чтобы научить игровой сервер отдавать файлы по протоколу HTTP.

Задание
Добавьте в игровой сервер возможность отдавать содержимое статических файлов, расположенных внутри заданного каталога.
В каталоге sprint2/problems/static_content/precode вы найдёте каталог static, содержащий файлы клиентской части игры. Передав путь к этому каталогу при старте сервера, вы сможете побродить по веб-страницам вашего сервера и даже запустить игру в браузере.
image

Вид игрового экрана. Пока здесь можно только вращать камеру вокруг собаки.
Входные данные
Путь к корневому каталогу со статическими файлами передаётся во втором параметре командной строки (первый параметр — путь к конфигурационному файлу игры). Например:

build/bin/game_server data/config.json static 

Выходные данные
GET- и HEAD-запросы, в которых URI-строка начинается не с /api/, должны интерпретироваться сервером как запросы статических файлов внутри соответствующего каталога.
Например, в теле ответа на запрос GET /images/image1.png HTTP/1.1 сервер должен отдать содержимое файла images/image1.png относительно каталога со статическими файлами. Ответ должен содержать следующие заголовки ответа:

    Content-Type — MIME-тип контента, зависящий от расширения файла.
    Content-Length — размер файла в байтах

Значение заголовка Content-Type должно зависеть от расширения запрашиваемого файла:

    .htm, .html: text/html
    .css: text/css
    .txt: text/plain
    .js: text/javascript
    .json: application/json
    .xml: application/xml
    .png: image/png
    .jpg, .jpe, .jpeg: image/jpeg
    .gif: image/gif
    .bmp: image/bmp
    .ico: image/vnd.microsoft.icon
    .tiff, .tif: image/tiff
    .svg, .svgz: image/svg+xml
    .mp3: audio/mpeg

Определение MIME-типа по расширению файла должно выполняться без учёта регистра символов. Например, для файлов image1.png и image2.PNg должен возвращаться заголовок Content-Type: image/png. 
Для файлов с пустым или неизвестным расширением заголовок Content-Type должен иметь значение application/octet-stream.
Если URI-строка запроса задаёт путь к каталогу в файловой системе, должно вернуться содержимое файла index.html из этого каталога. Например, следующие строки запроса ссылаются на файл index.html в корневой директории веб-сервера.

    GET / HTTP/1.1
    GET /index.html HTTP/1.1

Сервер должен трактовать URI-строку запроса как URL-encoded. Например, по запросу GET /hello%20world.txt HTTP/1.1 должно вернуться содержимое файла hello world.txt из корневого каталога веб-сервера. Предлагаем вам написать функцию URL-декодирования строки самостоятельно.
Требования к работе REST API сервера остаются прежними.
Обработка ошибок
Если URI-строка ссылается на несуществующий файл, сервер должен вернуть ответ со статус-кодом 404 Not Found и Content-Type: text/plain. Текст в ответе остаётся на ваше усмотрение.
Если результирующий путь к файлу оказался вне корневого каталога со статическими файлами, сервер должен вернуть ответ со статус-кодом 400 Bad Request и Content-Type: text/plain. Текст ошибки в ответе остаётся на ваше усмотрение.
Требования по обработке ошибок при работе с REST API не изменяются.
