Паттерны логирования
Логирование — важная задача, которая возникает в разных программах. В этом уроке разберём как архитектурные подходы к организации логирования, так и частности — как именно это лучше делать.
Как правило, программа пишет один лог-файл, в который сохраняется вся значимая информация о её работе. В логе отражаются события. Помимо самого факта возникновения события, желательно записать также время его возникновения, дополнительные обстоятельства, чтобы упростить дальнейшую работу по чтению лога.
Для записи лог-файла создаётся специальный программный объект — логгер.
Единственная задача логгера — писать лог. Знать что-то о программе, в которой он работает, необязательно. Программа должна сообщить логгеру о возникшем событии.
Давайте напишем простейший логгер.

#include <chrono>
#include <string_view>
#include <fstream>
#include <iostream>
#include <iomanip>

using namespace std::literals;

class Logger {
    static auto GetTimeStamp() {
        // C++20 позволяет напрямую выводить time_point в поток,
        // но на данный момент эта возможность есть не во всех популярных реализациях
        // стандартной библиотеки, поэтому воспользуемся функцией to_time_t
        // для форматирования момента времени.
        const auto now = std::chrono::system_clock::now();
        const auto t_c = std::chrono::system_clock::to_time_t(now);
        return std::put_time(std::gmtime(&t_c), "%F %T");
    }

public:
    Logger() {}

    void Log(std::string_view message) {
        log_file_ << GetTimeStamp() << ": "sv << message << std::endl;
    }

private:
    std::ofstream log_file_{"sample.log"s};
};

int main() {
    Logger log;
    log.Log("Program started"sv);

    int x, y;
    std::cout << "Enter x and y: ";
    std::cin >> x >> y;
    log.Log("User entered numbers "s + std::to_string(x) + " and "s + std::to_string(y));

    int sum = x + y;
    std::cout << "The sum of x and y is " << sum << std::endl;

    log.Log("The sum "s + std::to_string(sum) + " is computed"s);
}

В этом логгере, несмотря на его простоту, есть несколько проблем:

    Мы пишем лог-файл в текущую директорию.
    Каждый объект, пишущий лог, должен получать ссылку на объект Logger. Придётся добавлять его во всю иерархию конструкторов.
    Логировать неудобно: приходится все сущности вручную конвертировать в строку.
    Он не потокобезопасный.
    При каждом запуске лог-файл перезапишется.

Будем решать эти проблемы по очереди. Начнём с выбора правильной директории.

$ ls -l /var
drwxr-xr-x 2 root root  4096 Apr 15  2020 backups
...
drwxr-xr-x 3 root root  4096 Aug  1 13:22 log
...
drwxrwxrwt 2 root root  4096 Aug  1 13:25 tmp 

Способов всё-таки записать лог из программы несколько:

    Запустить программу от имени суперпользователя. Это небезопасно.
    Предварительно создать лог-файл вручную и изменить на него права, чтобы программа могла писать. Это неудобно.
    Внести изменение в систему — поменять права папки /var/log. Это ненадёжно.

Есть, однако, и стандартный способ обойти это препятствие — установить утилиту rsyslog. Она добавляет группу syslog и позволяет членам этой группы редактировать папку /var/log:

$ sudo apt install -y rsyslog
$ ls -l /var

...
drwxrwxr-x 1 root syslog 4096 Aug  1 13:22 log
...
drwxrwxrwt 2 root root   4096 Aug  1 13:25 tmp 

Затем можно добавить пользователя в группу syslog, чтобы он смог создавать файлы в нужной папке:

$ sudo usermod -a -G syslog rufus

# проверим, что можем создать лог-файл
$ touch /var/log/test_log 

После добавления группы, нужно переподключиться к серверу или перезагрузить машину, если соединение не удалённое.
Пишем лог из любого места программы
Вторая проблема состояла в том, что мы не хотели тянуть объект-логгер по всем классам программы. У неё тоже есть несколько очевидных решений, но, в отличие от предыдущей альтернативы из трёх вариантов, связанной с правами доступа к папке /var/log, они все имеют право на существование:

    Логируем через глобальную функцию.
    Создаём глобальную переменную — логгер.
    Делаем логгер объектом-одиночкой (singleton object).

Первые два решения тривиальны, разберём последнее. Оно заключается в том, что в класс добавляется статический метод GetInstance, возвращающий ссылку на единственный объект этого класса. Конструктор делается приватным, чтобы нельзя было случайно нарушить правило единственности.
❗ Паттерн Singleton иногда нежелателен. Он может приводить к лишним зависимостям в программе. В более масштабных приложениях объекты-одиночки могут ухудшить тестирование и анализ кода. Используйте Singleton’ы с умом.

class Logger {
    static auto GetTimeStamp() {
        // C++20 позволяет напрямую выводить time_point в поток,
        // но на данный момент эта возможность есть не во всех популярных реализациях
        // стандартной библиотеки, поэтому воспользуемся функцией 
        // to_time_t для форматирования момента времени.
        const auto now = std::chrono::system_clock::now();
        const auto t_c = std::chrono::system_clock::to_time_t(now);

        // %F используется для даты, а %T для времени. Оба значения будут 
        // выведены в формате ISO 8601. Полное описание всех плейсхолдеров:
        // https://en.cppreference.com/w/cpp/io/manip/put_time
        return std::put_time(std::gmtime(&t_c), "%F %T");
    }

    // конструктор теперь приватный
    Logger() = default;

public:
    // убираем конструктор копирования
    Logger(const Logger&) = delete;
    Logger& operator=(const Logger&) = delete;
    Logger(Logger&&) = delete;
    Logger& operator=(Logger&&) = delete;

    // получение ссылки на единственный объект
    static Logger& GetInstance() {
        static Logger obj;
        return obj;
    }

    void Log(std::string_view message) {
        log_file_ << GetTimeStamp() << ": "sv << message << std::endl;
    }

private:
    // для демонстрации пока оставим файл в текущей директории
    std::ofstream log_file_{"sample.log"s};
};

int main() {
    // логирование теперь делается иначе
    Logger::GetInstance().Log("Test logging"sv);
} 

При использовании статических объектов нужно быть осторожным. Разберёмся почему. Для начала ответьте на вопрос.
Когда будет сконструирован объект класса Logger?
Во время компиляции.

Он не будет сконструирован: логирование происходит без создания объекта.
Как вам задача?
Мы использовали локальную статическую переменную. Она создаст объект при первом входе в функцию GetInstance. 
Следующий вопрос, который возникает в связи с этим.
Когда и как объект логгера будет удалён?
Неправильный ответ
Последним действием программы.
Увы, после удаления статического объекта могут быть ещё действия. Например, удаления других объектов.

После удаления статического объекта, который был сконструирован после него.

Как вам задача?
Статические объекты удаляются в порядке, противоположном тому, как они завершили своё создание. Принцип логичный: если объект B ссылается на A, то B должен быть сконструирован после A, а удалён до. Это нужно, чтобы во время жизни B объект A постоянно существовал.
Теперь представьте, что какой-то объект логирует своё удаление. Логично разместить эту операцию в деструкторе:

class Server {
public:
...
    ~Server() {
        Logger::GetInstance().Log("Server stopped"sv);
    }
}; 

Такое решение опасно. Если предположить, что сервер конструируется внутри статического объекта, а логгер до этого не использовался, то логгер может быть сконструирован после сервера. В этом случае при работе деструктора возникнет неопределённое поведение: обращение к уже удалённому логгеру.
Чтобы этого не происходило, следуйте простым правилам:

    Не используйте статические объекты в деструкторах. Например, можно залогировать завершение работы перед командой остановки сервера, а не в деструкторе.
    Инициируйте создание логгера как можно раньше. Например, первой командой в main или в инициализаторе глобальной переменной.

Оба правила — не панацея. Например, логирование может быть вызвано не прямо в деструкторе, а опосредованно из него. Также никто не может дать гарантии, что объект инициализировался первым: порядок инициализации глобальных статических переменных из разных единиц трансляции не определён.
Удобство превыше всего
Вряд ли кто-то назовёт написание конструкций типа log.Log("User entered numbers "s + std::to_string(x) + " and "s + std::to_string(y)); удобным. Рассмотрим несколько альтернативных вариантов логирования:

    Через параметры функции:

log.Log("User entered numbers "s, x, " and "s, y); 

    Через возврат потока:

log.Log() << "User entered numbers "s << x << " and "s << y; 

    Через лямбда-функцию:

log.Log([&](std::ostream& ost) {
    ost << "User entered numbers "s << x << " and "s << y;
});

Второй способ, использующий возврат потока, наиболее прост в реализации:

class Logger {
   ...

public:
    std::ostream& Log() {
        return log_file_;
    }

private:
    // для демонстрации пока оставим файл в текущей директории

int main() {
    int x, y;

У него ряд недостатков. Во-первых, мы не можем контролировать завершающий текст. Пользователь должен сам вывести std::endl. Во-вторых, этот способ не страхует от того, что пользователь сохранит поток и будет использовать его не по назначению, что нарушит целостность лог-файла. Оба недостатка можно устранить, если вернуть не поток напрямую, а специальный прокси-объект, который фактически выводит в своём деструкторе. Этот способ применяется в библиотеке Boost.Log. Её мы разберём в следующих уроках.
Из приведённых трёх способов логирования наибольшую гибкость даёт третий — через лямбда-функцию:

class Logger {
   ...

public:
    void LogMessage(std::string_view message) {
        Log([message](std::ostream& out) {
            out << message;
        });
    }

    template<class Func>
    void Log(Func&& cb) {
        log_file_ << GetTimeStamp() << ": "sv ;
        cb(log_file_);
        log_file_ << std::endl;
    }

private:
    // для демонстрации пока оставим файл в текущей директории
    std::ofstream log_file_{"sample.log"s};
};

int main() {
    int x, y;
    std::cin >> x >> y;

    Logger::GetInstance().Log([x, y](std::ostream& o) {
        o << "User entered numbers "sv << x << " and "sv << y;
    });
} 

Его основной недостаток в том, что логировать всё же не так удобно: нужно каждый раз писать список параметров лямбды и указывать переменные связывания. Это можно упростить, введя макрос:

#define LOG(X) Logger::GetInstance().Log([&](std::ostream& o) { o << X; })

int main() {
    int x, y;
    std::cin >> x >> y;

    LOG("User entered numbers "sv << x << " and "sv << y);
} 

Но макросы тоже имеют недостатки. В задаче этого урока вам будет предложено реализовать способ логирования через variadic template.
Проблемы многопоточности
Логирование нужно не только однопоточным программам, в которых указанный выше способ прекрасно работает, но и многопоточным.
При записи из разных потоков могут возникнуть проблемы:

int main() {
    std::thread thread1([](){
        for(int i = 0; i < 1000; ++i) {
            LogIndexInThread(1, i);
        }
    });
    std::thread thread2([](){
        for(int i = 0; i < 1000; ++i) {
            LogIndexInThread(2, i);
        }
    });
    
    thread1.join();
    thread2.join();
}

osyncstream введён в C++20 специально для решения проблем многопоточности при выводе. Вы уже встречались с ним в первом спринте.
Как вам задача?

#include <syncstream>

class Logger {
...
public:
    template<class Func>
    void Log(Func&& cb) {
        std::osyncstream sync_stream(log_file_);

        cb(sync_stream);
        sync_stream << std::endl;

К сожалению, на данный момент osyncstream не реализован в библиотеке libc++, которая используется по умолчанию в clang.
Можно обойтись и без него, использовав std::mutex, который доступен уже во всех компиляторах, поддерживающих C++11:

class Logger {
...
public:
    template<class Func>
    void Log(Func&& cb) {
        std::lock_guard lock(m_);

        log_file_ << GetTimeStamp() << ": "sv ;
        cb(log_file_);
        log_file_ << std::endl;
    }

private:
...

    std::mutex m_;
};
 

С обновлённым логгером никаких проблем не возникает. Запустите код по ссылке и убедитесь в этом.
Конечно, правильный результат ещё не значит, что программа работает верно. Это в особенности касается многопоточных приложений. Но в данном случае работа программы правильная. Правда, есть изъян: пока один поток не закончит логирование, остальные будут ждать. Особенно проблемно, если логирование производится через лямбда-функцию, а в неё встроена какая-то логика. Более совершенный логгер при запросе логирования не отправит данные в поток, а быстро сохранит их в контейнере. В идеале это будет lock-free контейнер. Затем асинхронно запишет данные в файл.
Открываем правильно
Мы переходим к последнему отмеченному недостатку логгера — обнулению логов при каждом перезапуске. Хоть недостаток и последний, но победить его довольно просто. А как — ответьте сами.

Как вам задача?
С параметром std::ios::app содержимое файла будет дописываться, а не перезаписываться. Вопрос только, как долго. Ведь программа может работать годами, и лог файл в этом случае достигнет очень внушительных размеров. Анализировать получившийся многогигабайтный файл будет очень сложно. Для решения этой проблемы файл лога периодически меняют. При этом используют два признака того, что лог пора заменить:

    Размер. Переходить к новому файлу, если текущий достиг определённого размера.
    Дата изменения. Разделять логи по дням или месяцам.

Первый признак позволит получить равномерные лог-файлы и гарантирует, что они не будут слишком большими. Второй обеспечивает удобную навигацию. Вы сможете быстро понять, где искать нужное событие. Идеальным вариантом было бы комбинирование первого и второго признака.

Задание
Возьмите заготовку задачи из папки sprint2/problems/logger.
Разработайте логгер, который использует variadic template для логирования. Процедура должна выглядеть так:

Logger::GetInstance().Log("User entered numbers "sv, x, " and "sv, y); 

Метод Log должен принимать произвольное количество аргументов, и каждый из них он должен записать в лог. Каждый аргумент записывается простым выводом в поток. Предполагается, что пользователь определил для нужных типов operator << (std::ostream&, const T&). Перед аргументами выводится временная метка и символы : . Временную метку можно получить методом GetTimeStamp(), имеющимся в заготовке. Такой вывод мог получиться в примере выше:

Для тестирования добавьте дополнительный метод SetTimestamp, который принимает один параметр — временну́ю метку. После вызова этого метода логгер должен использовать в качестве текущего времени переданное значение, а не system_clock::now().
Логгер должен записывать лог в файл /var/log/sample_log_YYYY_MM_DD.log, в котором вместо MM нужно подставить текущий месяц, вместо DD — текущий день месяца, а вместо YYYY — текущий год. День и месяц должны при необходимости иметь ведущие нули. При выводе следует полагаться на дату, переданную в SetTimestamp.
Если в процессе логирования изменилась дата, то нужно открыть другой файл.
Логгер должен быть потокобезопасным и корректно обрабатывать логирование из разных потоков. Возможна также ситуация смены даты во время логирования, её нужно учесть.
